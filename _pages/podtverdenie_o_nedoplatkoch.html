Через портал eformulare.socpoist.sk на macOS. Авторизация проходит через мобильное приложение по NFC с картой побыта (BOK + сертификаты KEP). Подпись - с помощью читачки.

Предусловия

  1. Физический USB smart-card ридер (Čítačka)
  2. Карта побыта с KEP-сертификатом.
  3. Установлено приложение eIDENTITA на смартфон
  4. Установлены:
    • eID klient
    • D.Signer/XAdES
    • Расширение D.Bridge 2 для Chrome https://chromewebstore.google.com/detail/dbridge-2/fngbdhimbgbonhlibfmiemipheabfdmj
  5. Сайт открывается только если вы в Словакии (либо у вас словацкий IP-адрес)

Порядок действий

Убедиться что eID klient запущен.

Авторизоваться на slovensko.sk. Это можно сделать через мобильное приложение eIDENTITA с помощь NFC-авторизации. Для подписи понадобится читачка.

Вставить карту в ридер (чипом вверх) и проверить что она видна системе. (использование читачки.html)

Перейти на https://eformulare.socpoist.sk. На странице выбрать форму «Žiadosť o vydanie potvrdenia o nedoplatkoch na sociálnom poistení» → «Detail služby».

На экране «Výber subjektu pre zastupovanie» — выбрать нужный субъект (ИП) и нажать «Prihlásiť sa».

Проверить данные заявителя (Názov, IČO, DIČ) → нажать «Pokračovať na podpísanie».

Следующие шаги нужно выполнит достаточно быстро, чтобы криптографические программы не потеряли сессию. На странице «Podpísanie» нажать «Podpísať formulár».

Появится прелоадер «Čakajte prosím...» — идёт запуск D.Signer через D.Launcher.

D.Launcher 2 спросит подтверждения запуска D.Signer/XAdES Java → «Run».

Откроется диалог «Nastavenia spôsobu prístupu k certifikátom». Нажать зелёную круглую стрелку (refresh) справа от выпадашки — в поле «Slot» должны появиться:

  • #1 ffffffff (Sig_ZEP) — KEP (квалифицированный), нужен для Sociálnej poisťovne
  • #2 ffffffff (Sig_EP) — обычный электронный подпис
Выделить строку #1 Sig_ZEP → кнопка Ďalej станет активной → нажать.

Откроется окно D.Signer/XAdES Java с текстом документа («Žiadosť o vydanie potvrdenia...»).

Нажать Podpísať внизу справа.

Появится окно «Výber certifikátu» — выбрать сертификат «SVK eID ACA2» → «OK».

Во всплывающем окне ввести шестизначный код.

Статус меняется на зелёный Dokument je podpísaný, кнопка OK становится активной → нажать OK.

Открывается страница «Sumarizácia a odoslanie» → нажать ODOSLAŤ PODANIE.

Появляется диалог подтверждения отправки → нажать Zatvoriť.

Главное — делать всё быстро. Сессия на портале живёт 10–15 минут, если зависать в D.Signer — умрёт.

Решение проблем

Диагностика ридера и карты

Советы по диагностике USB smart-card ридера — в отдельном файле: использование читачки.html

Процессы, которые должны быть живы при подписи

ps aux | grep -iE "eid_klient|d\.signer|d\.launcher|dbridge" | grep -v grep

Ожидаемо видеть:

  • /Applications/eID_klient.app/Contents/MacOS/eID_Client
  • D.Launcher 2.app/.../dLauncher2
  • dBridge2Nm (расширение браузера)
  • Java-процесс sk.ditec.dbridge2.DBridgeServerApp (запускается во время подписи)

Пустой список слотов

Проблема в том, что программа не видит карту побыта даже после нажатия зеленой круговой стрелки.

2. Запущен ли eID klient в фоне?

ps aux | grep -i eID_Client | grep -v grep

Должна быть строка /Applications/eID_klient.app/Contents/MacOS/eID_Client. Если процесса нет — запусти eID klient из Applications (он стоит в автозапуске, но после reboot может ещё не подняться).

Окна у eID klient нет — это нормально.

3. macOS видит ридер и карту на низком уровне?

system_profiler SPSmartCardsDataType | head -5

Должен быть виден ридер с ATR карты. Подробно все состояния и что с ними делать — использование читачки.html.

4. Конфликт с системным CCID macOS (самая частая причина на Sonoma/Sequoia)

Начиная с Ventura macOS сама подхватывает карту через com.apple.CryptoTokenKit.pivtoken и eID klient получает карту «занятой». Лечение:

sudo defaults write /Library/Preferences/com.apple.security.smartcard DisabledTokens -array com.apple.CryptoTokenKit.pivtoken

Перезагрузить мак. После этого eID klient должен получить эксклюзивный доступ.

7. Кэш/залипший процесс

ps aux | grep -iE "d\.signer|d\.launcher|eid_klient"

Если что-то висит — убить, перезапустить последовательность с пункта 1 happy path.

Прелоадер на веб-странице не исчезает после подписания

При этом окно D.Signer/XAdES Java зависает (Баг Ditec).

Сессия на eformulare.socpoist.sk истекает по неактивности минут за 10–15. Если долго возиться с подписанием — сессия умрёт, и даже успешная подпись не дойдёт до веб-страницы.

Поможет перезагрузка компьютера. Все шаги нужно сделать быстро.

Симптомы:

  • В окне D.Signer нажал Podpísať, выбрал сертификат, ввёл KEP PIN — всё прошло успешно.
  • Возвращаешься в основное окно D.Signer — оно висит, кнопки (OK, Storno) не реагируют.
  • Прелоадер на странице формы крутится бесконечно.
  • ps aux показывает Java-процесс с 0% CPU.
  • lsof -iTCP -sTCP:LISTEN | grep java пусто — WebSocket-сервер внутри Java мёртв.

Настоящая причина (найдена через sample <pid> 3 -mayDie):

Стек главного Java-потока:

JVM_Halt → vm_exit → exit → __cxa_finalize_ranges
  → lib_fini() (in libPkcs11.dylib) + 60
    → std::thread::join()
      → _pthread_join  ← висит вечно

Java корректно завершил подпись и вызвал System.exit(0). Но при выходе JVM вызывает финализаторы нативных библиотек, и libPkcs11.dylib от Ditec застревает в lib_fini(): его фоновый поток (для общения со смарт-картой через CryptoTokenKit) не отвечает на shutdown, pthread_join ждёт бесконечно.

Последствие: процесс Java не завершается → stdout-буферы native messaging не флашатся → Chrome не получает ответ → прелоадер.

Почему это происходит именно на Apple Silicon:

JRE, который поставляется с D.Launcher 2, — Intel-only (Mach-O 64-bit executable x86_64), идёт через Rosetta 2. Проверить:

file /Users/user/.ditec/dlauncher2/products/sk.ditec.djre/*/bin/java
# Mach-O 64-bit executable x86_64

libPkcs11.dylib — universal (x86_64+arm64), но раз Java x86_64, грузится именно Intel-слайс. Связка «x86_64 JVM через Rosetta + нативные потоки к CTK» стабильно ловит дедлок в dyld-финализаторах. Это баг Ditec — они не поставляют нативный ARM JRE.
