# Клиентский поиск по гайду

**Дата:** 2026-04-26
**Статус:** утверждено пользователем, готово к плану реализации

## Цель

Оживить уже существующее поле поиска в шапке сайта (`_includes/header.html`)
полнотекстовым поиском по всем гайдам. Сайт остаётся статическим
(Jekyll → GitHub Pages), весь поиск выполняется в браузере.

## UX

- Поле поиска уже размещено в шапке: `_includes/header.html:8-15`,
  с подсказкой `<kbd>⌘K</kbd>`.
- **Хоткей:** `⌘K` (mac) / `Ctrl+K` (остальные) фокусирует поле и
  выделяет текущий ввод. Перехват хоткея отключён, если фокус
  находится в `input/textarea/contenteditable`.
- **`Esc`** убирает фокус и закрывает дроплист.
- **`↑ / ↓`** навигация по результатам, **`Enter`** — переход.
- **Дроплист** позиционируется под полем (`position: absolute`
  относительно `.site-search`), z-index поверх контента. Открывается
  при непустом запросе и наличии результатов. Закрывается по `Esc`,
  клику вне, или после клика по результату.
- **Максимум 8 результатов.**
- **Карточка результата:** заголовок страницы → matched heading
  (если совпадение попало в `headings`) → сниппет ~140 символов
  вокруг первого совпадения по `content` с `<mark>`-подсветкой
  искомых слов → URL мелким шрифтом. Если совпадение только по
  `title` — сниппет не показываем.
- **Пустой результат:** показываем строку «Ничего не найдено»
  внутри дроплиста, дроплист остаётся открытым.
- **Состояния загрузки:**
  - До завершения `fetch('/search.json')` — «Загрузка…».
  - При ошибке — «Поиск временно недоступен», `console.error` в логе.

## Архитектура

Три новых артефакта:

### 1. `search.json` (Jekyll-шаблон)

Файл в корне репо с front matter:

```yaml
---
permalink: /search.json
layout: null
---
```

Тело — Liquid-шаблон, итерирующий `site.pages`. Для каждой страницы
выдаёт объект `{title, url, headings, content}`:

- `title` — `page.title`
- `url` — `page.url | relative_url`
- `headings` — массив строк, извлечённых из `<h2>` и `<h3>` контента
  (через regex по `page.content` или просто `strip_html` от
  отфильтрованных частей; уточнить в плане реализации)
- `content` — `page.content | strip_html | strip_newlines |
  normalize_whitespace`

**Исключения из индекса:**

- страницы с `nav: false` в front matter;
- партиалы из `_pages/podtverdenie_o_nedoplatkoch/` (служебные
  фрагменты, не самостоятельные страницы).

Опциональный флаг `search: false` для точечного исключения **не
вводим** (YAGNI — добавим, если появится конкретный кейс).

Ожидаемый размер для 19 страниц: 80–150 КБ несжатых,
~25–40 КБ gzip.

### 2. `assets/js/search.js`

Модуль, ~150 строк. Поведение:

- На `DOMContentLoaded` находит `input[type=search]` в
  `.site-search`, оборачивает его в относительно
  позиционированный контейнер (если ещё не обёрнут), создаёт
  пустой `<div class="search-results">` рядом.
- На первый `focus` поля делает `fetch('/search.json')`, строит
  Fuse-индекс, кэширует ссылку в модульной переменной.
- На каждый `input`-event фильтрует и рендерит дроплист.
- Реализует клавиатурную навигацию (`↑/↓/Enter/Esc`) и закрытие по
  клику вне.
- Глобальный обработчик `keydown` для `⌘K / Ctrl+K`.

**Конфигурация Fuse.js:**

```js
{
  keys: [
    { name: 'title',    weight: 3 },
    { name: 'headings', weight: 2 },
    { name: 'content',  weight: 1 }
  ],
  threshold: 0.3,
  ignoreLocation: true,
  minMatchCharLength: 2,
  includeMatches: true
}
```

`includeMatches: true` нужен для сниппета и подсветки совпадений.

### 3. Стили для дроплиста

Минимальный CSS, ~30–50 строк:

- `.search-results` — абсолютно позиционированный контейнер под
  полем, `display: none` по умолчанию.
- `.search-results.is-open` — показываем.
- `.search-result` — одна карточка; `:hover` и `.is-active`
  (выделение клавиатурой) подсвечиваются.
- `.search-result__title`, `.search-result__heading`,
  `.search-result__snippet`, `.search-result__url` — типографика
  карточки.
- `mark` внутри сниппета — фон акцентного цвета.

Перед реализацией свериться с тем, как у проекта организован CSS
(один main-файл, или отдельные `assets/css/*.css`), и положить
стили туда же. План решит этот вопрос точечно.

### Подключение Fuse.js

Через CDN тегом `<script defer src="https://cdn.jsdelivr.net/npm/fuse.js@7/dist/fuse.min.js"></script>`
в `_includes/head.html` (или где сейчас сидят сторонние скрипты).
~10 КБ gzip. Версию пинуем мажорно (`@7`).

## Поток данных

```
build time:
  Jekyll → search.json (статический файл в _site)

runtime:
  user focuses input
    → fetch /search.json
    → new Fuse(data, options)
    → cached for session

  user types
    → fuse.search(query, { limit: 8 })
    → render dropdown with snippets

  user navigates ↑/↓ → highlight item
  user presses Enter → window.location = result.url
  user presses Esc / clicks outside → close dropdown
```

## Тестирование

Статический сайт без тест-фреймворка. Проверка ручная по чек-листу:

1. `bundle exec jekyll serve` → `/search.json` отдаётся, валидный
   JSON, 19 записей (или меньше после фильтрации `nav: false`).
2. `⌘K` фокусирует поле; ввод «транс» показывает «Транспорт»
   в первой позиции.
3. `↑/↓/Enter` навигация работает.
4. Поиск по тексту внутри страницы (например слово «MHD») ведёт на
   правильную страницу.
5. `Esc` закрывает дроплист, клик вне закрывает.
6. В DevTools во вкладке Network `/search.json` грузится только
   после первого фокуса (lazy-load).
7. Опечатка («трнаспорт») всё ещё находит страницу «Транспорт»
   благодаря fuzzy-поиску.

## Out of scope

- Сниппеты с подсветкой по нескольким совпадениям одновременно
  (рисуем только первое).
- Морфологический поиск/стеммер для русского — Fuse fuzzy достаточен
  для гайда из 19 страниц.
- Аналитика поисковых запросов.
- Поиск по якорям (`#h2-id`) внутри страниц — индекс работает на
  уровне страницы.
- Подсветка совпадения на целевой странице после перехода.
